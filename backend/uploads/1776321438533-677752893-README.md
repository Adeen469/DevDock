# QA Engineer (Automated Multi-Testing Platform)

Complete upload-and-test platform implemented under `QA-engineer` only.

## What It Supports

- Project upload from:
  - ZIP file
  - Single/multiple files
  - Folder upload (Google Drive-like webkitdirectory flow)
- Testing modes:
  - Unit and white-box testing via Jest
  - API integration testing via Postman collection + Newman
  - UI/black-box testing via Playwright
  - Performance testing via Apache JMeter
- One-click CI workflow:
  - `.github/workflows/test-suite.yml` inside `QA-engineer`
  - Runs all test stages and uploads artifacts

## Folder Layout

- `backend`: Express API, MySQL persistence, test orchestration services
- `frontend`: React (Vite) command center UI
- `.github/workflows/test-suite.yml`: central CI command workflow
- `docker-compose.yml`: MySQL local stack

## Backend Setup

1. Start MySQL:

```bash
docker compose up -d
```

2. Configure env:

```bash
cd QA-engineer/backend
cp .env.example .env
```

3. Install dependencies and initialize schema:

```bash
npm install
npm run db:init
```

4. Start backend:

```bash
npm run dev
```

Backend default URL: `http://localhost:4200`

## Frontend Setup

```bash
cd QA-engineer/frontend
npm install
npm run dev
```

Frontend default URL: `http://localhost:5173`

## Frontend Features

- **Project Upload**: Google Drive-style folder upload, ZIP archives, or individual files
- **Database Setup Wizard**: Configure MySQL cluster URL/credentials and test connection before running tests
- **Real-time Test Output**: Live log streaming via Server-Sent Events (SSE) while tests run
- **Run Details Modal**: Click any run in history to view:
  - Live test output logs
  - Current run status and timestamps
  - Download zipped test reports button
- **Auto-refresh**: Run history updates every 3 seconds to show latest test status

## API Endpoints

- `POST /api/projects/upload` - Upload ZIP/files/folder content
- `GET /api/projects` - List uploaded projects
- `POST /api/runs/project/:projectId` - Start full or partial test suite
- `GET /api/runs` - List previous runs
- `GET /api/runs/:runId/log` - Retrieve complete run log
- `GET /api/runs/:runId/logs/stream` - Stream live run logs (Server-Sent Events)
- `GET /api/runs/:runId/artifacts/download` - Download zipped test reports
- `POST /api/health/db` - Test database connection with provided credentials
- `GET /api/health` - Health check

## CI/CD Central Command

Workflow files:
- `/.github/workflows/qa-engineer-test-suite.yml` (active in GitHub Actions)
- `QA-engineer/.github/workflows/test-suite.yml` (local project copy)

Stages:
1. Jest unit/white-box tests
2. Newman API integration tests
3. Playwright UI/black-box tests
4. JMeter performance tests
5. Artifact upload (`ci-reports`)

## Notes

- API keys are not required for local/CI-free execution.
- If you later need Postman Cloud fetch, store `POSTMAN_API_KEY` as a secret and switch Newman command accordingly.
- The orchestrator generates fallback Postman, Playwright, and JMeter assets at run-time when triggering tests on uploaded projects.
- Local runs use `ALLOW_MISSING_TOOLS=true` by default, so unavailable heavy tools (like JMeter) are marked skipped instead of hard-failing.
- CI workflow sets `ALLOW_MISSING_TOOLS=false`, so every configured tool must execute successfully in pipeline runs.
- For local full performance execution, install JMeter and ensure `jmeter` is available in your PATH.
